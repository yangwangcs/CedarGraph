# CedarGraph 分布式架构分析报告

## 1. 分布式能力评估

### 1.1 架构概览

| 组件 | 状态 | 说明 |
|------|------|------|
| **MetaD** (元数据服务) | ✅ 已实现 | 3节点Raft集群，管理Space/Partition/Node |
| **StorageD** (存储节点) | ⚠️ 部分实现 | 分区管理、心跳上报、2PC支持 |
| **客户端SDK** | ⚠️ 部分实现 | 路由、连接池、事务支持 |
| **gRPC传输** | ✅ 已实现 | 节点间通信、TLS支持 |

### 1.2 分布式事务支持

```
┌─────────────────────────────────────────────────────────────┐
│                    分布式事务分层模型                          │
├─────────────────────────────────────────────────────────────┤
│ Layer 1: 单分区事务 (SinglePartition)                        │
│          → 无需协调，直接本地执行                              │
├─────────────────────────────────────────────────────────────┤
│ Layer 2: 同时序范围跨分区 (SameTemporalRange)                 │
│          → 轻量协调，共享时间戳                                │
├─────────────────────────────────────────────────────────────┤
│ Layer 3: 跨时序范围 (CrossTemporalRange)                      │
│          → 完整2PC协议                                        │
├─────────────────────────────────────────────────────────────┤
│ Layer 4: 确定性批量事务 (Deterministic)                        │
│          → Calvin风格，预排序执行                              │
└─────────────────────────────────────────────────────────────┘
```

**2PC状态机:**
- `kStarted` → `kPreparing` → `kPrepared` → `kCommitting` → `kCommitted`
- 支持 `kAborting` → `kAborted` 回滚路径

## 2. 数据落盘与持久化

### 2.1 多层持久化架构

```
┌─────────────────────────────────────────────────────────────┐
│                      写入路径                                  │
├─────────────────────────────────────────────────────────────┤
│ 1. WAL (Write Ahead Log)                                     │
│    - 默认启用 (enable_wal = true)                             │
│    - 支持分片WAL (Sharded WAL) 高并发优化                      │
│    - SyncWAL() 强制刷盘                                       │
├─────────────────────────────────────────────────────────────┤
│ 2. MemTable (内存表)                                          │
│    - 默认4MB阈值 (write_buffer_size)                          │
│    - 跳表实现 (O(log n) 插入/查询)                            │
├─────────────────────────────────────────────────────────────┤
│ 3. Immutable MemTable                                         │
│    - 冻结的MemTable，等待刷盘                                  │
├─────────────────────────────────────────────────────────────┤
│ 4. SST文件 (磁盘)                                             │
│    - Zone-Columnar格式 (256KB Block，稀疏索引)                 │
│    - 累积Flush生成大文件 (默认64MB)                            │
│    - 布隆过滤器加速查询                                        │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 Raft日志持久化

## 3. 图分割策略 (Partitioning)

### 3.1 分区键设计

```cpp
// CedarKey 结构 (38字节)
┌─────────────────────────────────────────────────────────────┐
│  entity_id (8) │ type(1) │ col(2) │ ts(8) │ tgt(8) │ part(2) │
└─────────────────────────────────────────────────────────────┘
         ↓                                              ↓
    实体唯一标识                                   分区ID (16-bit)
```

### 3.2 分区策略

```cpp
// 当前实现: entity_id % 65536
static inline uint16_t ComputePartition(uint64_t entity_id) {
    return static_cast<uint16_t>(entity_id);  // 取低16位
}
```

**分区特性:**
- **最大分区数**: 65,536 (2^16)
- **默认分区数**: 128 (可配置)
- **分片键**: entity_id (确保同一实体所有版本在同一分区)
- **边分区**: 按 src_id 分区 (出边) / dst_id 分区 (入边)

## 5. 可靠性保障

### 5.1 故障检测

```
┌─────────────────────────────────────────────────────────────┐
│                    心跳机制                                    │
├─────────────────────────────────────────────────────────────┤
│ StorageD → MetaD                                             │
│   - 间隔: 5秒 (heartbeat_interval)                            │
│   - 上报: CPU/内存/磁盘使用率, QPS, 延迟, 分区状态              │
│                                                              │
│ MetaD 心跳检查                                                │
│   - 超时: 10秒 (heartbeat_timeout_sec)                        │
│   - 检查间隔: 5秒                                             │
│   - 节点状态: Online/Offline/Suspected                        │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 故障恢复

| 场景 | 机制 | 状态 |
|------|------|------|
| **MetaD节点故障** | Raft自动选举新Leader | ✅ 已实现 |
| **StorageD节点故障** | 心跳超时，标记Offline | ⚠️ 部分实现 |
| **Partition Leader故障** | 重新选举Leader | ⚠️ 部分实现 |
| **事务恢复** | 崩溃恢复，回查2PC状态 | ⚠️ 部分实现 |

### 5.3 数据一致性

```
┌─────────────────────────────────────────────────────────────┐
│                    一致性级别                                  │
├─────────────────────────────────────────────────────────────┤
│ kStrict    (严格一致性)  → 读取看到所有已提交写入               │
```

## 6. 总结与建议

### 6.1 当前状态评估

| 能力 | 完成度 | 关键缺陷 |
|------|--------|----------|
| **分布式部署** | 70% | StorageD副本同步需完善 |
| **数据持久化** | 85% | WAL分片需优化 |
| **图分割** | 80% | 仅支持哈希分区，无范围分区 |
| **多副本** | 60% | 故障转移自动化程度不足 |
| **可靠性** | 65% | 缺少完善的故障恢复测试 |

### 6.2 生产就绪检查清单

- [ ] **副本同步**: 实现存储层Raft或主从复制
- [ ] **自动故障转移**: StorageD Leader选举自动化
- [ ] **数据重平衡**: 分区迁移工具完善
- [ ] **监控告警**: 完善的Metrics和告警机制
- [ ] **混沌测试**: 故障注入测试覆盖
- [ ] **备份恢复**: 全量/增量备份机制
- [ ] **安全**: 认证、授权、加密传输

### 6.3 结论

**CedarGraph当前状态**: 
- ✅ **可用作单机/单分区图数据库**
- ⚠️ **分布式模式处于Alpha阶段**，需要进一步完善副本同步和故障恢复机制
- 📋 **建议**: 当前适合开发测试环境，生产环境使用需完成上述检查清单
