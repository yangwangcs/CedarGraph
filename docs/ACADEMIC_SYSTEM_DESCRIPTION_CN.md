# CedarGraph-Core 论文式系统说明

## 摘要

CedarGraph-Core 面向分布式时态图数据管理场景。时态图中的顶点和边会随时间形成多个版本，系统不仅需要回答当前状态查询，还需要回答历史快照、时间范围和变化过程查询。在分布式环境中，这类负载会同时放大事务协调、冲突检测、历史版本扫描、分区质量和计算视图一致性问题。

本文说明 CedarGraph-Core 的系统方法：用 LND-OCC 将事务按参与分区和时间范围分层，减少不必要的分布式协调；用 TW-CD 将时间窗口纳入冲突检测，降低 key-only OCC 的保守 abort；用 CedarKey、LSM、WAL、SST 和版本折叠组织时态数据；用 GCN/TMV 为频繁遍历维护多版本计算视图。本文只描述当前仓库中的工程事实和可验证设计，不把历史审计数字写成生产承诺。

## 1. 问题定义

分布式时态图系统的问题来自图结构、时间版本和分布式故障三者叠加。

第一，图更新经常跨越顶点、出边、入边、索引和分区。如果所有写入都走完整两阶段提交，单分区事务也会承担网络往返和协调日志成本。

第二，时态图中的冲突不能只按 key 判断。两个事务访问同一实体但时间窗口不重叠时，很多语义下可以并发执行。传统 OCC 容易把这类访问误判为冲突。

第三，LSM-Tree 适合写入，但时间旅行查询需要在多版本中找到快照可见状态。若 key 编码和版本折叠设计不足，读路径会扫描大量无效历史版本。

第四，分区器不能只做哈希均衡。时态图需要同时维护时间局部性、拓扑局部性和负载均衡，否则会出现跨分区遍历过多或热点分区长期过载。

第五，近实时图计算不能每次都从存储层回扫。计算节点可以维护多版本视图，但必须处理 CDC、watermark、缓存失效和远程补读。

## 2. 方法

### 2.1 LND-OCC

LND-OCC 将事务划分为四类路径。

| 层级 | 事务 | 路径 |
|---|---|---|
| Layer 1 | 单分区事务 | 本地 OCC 验证与提交 |
| Layer 2 | 同时间范围跨分区事务 | 轻量 prepare/commit |
| Layer 3 | 跨时间范围跨分区事务 | 完整 2PC |
| Layer 4 | 批处理事务 | 预留确定性批处理接口 |

该方法的核心是“按必要性协调”。单分区写入不应因为系统支持分布式事务而被迫进入分布式路径；复杂跨分区写入则仍由可恢复提交协议保护。

### 2.2 TW-CD

TW-CD 的冲突检测分两步：

1. 查询与当前事务时间窗口重叠的活跃事务。
2. 只在候选事务中执行 key 级读写/写写冲突判断。

因此，不重叠时间窗口上的访问可以绕过不必要 abort。事务提交或中止后必须注销窗口并清理 key 索引，这是长时间运行正确性的关键不变量。

### 2.3 CedarKey 与 LSM 时态布局

CedarKey 是固定长度时态 key，编码实体、时间、目标实体、列、序列、实体类型、标记和分区。时间戳使用有利于快照读取的排序方式，使最新可见版本更容易定位。存储层通过 WAL、MemTable、Immutable MemTable、SST、Bloom Filter、Blob 文件和 compaction 管理多版本数据。

### 2.4 GCN/TMV

GCN 维护 TMV 多版本计算视图。写入事件通过 CDC 或事件应用路径进入计算层；查询优先命中本地视图，miss 时通过 Scatter-Gather 或回源存储补齐。该层用于降低高频遍历延迟，而不是替代持久化存储。

## 3. 系统运行机制

写入流程：

1. GraphD 解析请求并确定读写集、时间窗口和参与分区。
2. MetaD 提供 schema、分区和服务位置。
3. LND-OCC 选择提交路径。
4. TW-CD 判断是否存在时态冲突。
5. StorageD 写 WAL 和 MemTable，并通过 Raft 复制需要复制的状态。
6. 后台 flush/compaction 生成 SST，事件应用到 GCN/TMV。

时间旅行查询流程：

1. 查询层生成快照时间或时间范围。
2. 路由层定位相关分区。
3. 存储层通过 CedarKey 和版本折叠查找可见版本。
4. 遍历类查询优先使用 GCN/TMV；无法命中时回源存储。

## 4. 预期实验

| 实验 | 指标 | 预期观察 |
|---|---|---|
| LND-OCC vs 全量 2PC | 延迟、RPC 次数、吞吐 | 单分区事务协调成本下降 |
| TW-CD vs key-only OCC | abort 率、验证耗时 | 时间不重叠负载 abort 率下降 |
| CedarKey 快照读取 | SST 访问、P95/P99 | 历史版本无效扫描减少 |
| 分区策略对比 | 跨分区边比例、负载方差 | 时态/拓扑感知策略改善局部性 |
| GCN/TMV | 本地命中率、遍历延迟 | 视图命中时延迟低于回源存储 |
| 故障恢复 | 数据一致性、恢复时间 | Raft 与事务日志保证可恢复 |

## 5. 验证边界

本仓库包含构建、单元测试、烟测、部署静态检查、TLS/mTLS 检查和故障注入脚本。它们能证明当前代码在本地或 CI 环境下达到候选质量，但真实生产上线仍需要在目标环境中验证证书、Secret、网络、持久卷、监控、备份恢复、回滚和长时间压力。
