## 第二部分: 可行性分析

### 2.1 为什么需要图分析层?

| 维度 | 分析 |
|------|------|
| **用户需求** | 现代图数据库标配 (Neo4j GDS, TigerGraph GSQL, NebulaGraph Algorithm) |
| **竞争优势** | 没有分析能力的图数据库 = "高级KV存储" |
| **技术趋势** | 图神经网络(GNN)兴起，需要图特征支持 |
| **商业机会** | 实时图分析是高价值场景 |

### 2.2 可行性评估

#### ✅ 有利因素

1. **存储层就绪**
   - LSM引擎支持高效扫描
   - MVCC支持快照读取 (分析不影响写入)
   - 分区机制支持并行处理

2. **计算资源可用**
   - 3节点集群可扩展
   - 内存充足 (可配置缓存)

3. **事务层可复用**
   - 快照隔离适合分析 (一致视图)
   - 分区信息可用于任务分发

#### ❌ 挑战因素

1. **资源竞争**
   - 分析计算可能影响在线事务
   - 需要资源隔离机制

2. **计算模型选择**
   - 内存计算 vs 磁盘迭代
   - 同步计算 vs 异步计算
   - 全图计算 vs 子图计算

3. **算法复杂度**
   - PageRank等算法需要多轮迭代
   - 大图可能无法完全放入内存

### 3.2 组件详细设计

#### 3.2.1 嵌入式分析引擎 (Embedded Analytics)

```cpp
// 适用于轻量级、低延迟的分析查询
// 执行位置: 存储节点本地

namespace cedar::analytics::embedded {

// 路径算法
class PathAlgorithms {
 public:
  // 最短路径 (Dijkstra/BFS)
  Result<std::vector<Path>> ShortestPath(
      VertexId from, 
      VertexId to,
      ShortestPathOptions options);
  
  // 所有最短路径 (Yen算法)
  Result<std::vector<Path>> KShortestPaths(
      VertexId from,
      VertexId to, 
      size_t k);
  
  // 全对最短路径 (Floyd-Warshall, 小图)
  Result<Matrix> AllPairsShortestPaths(const Subgraph& subgraph);
};

// 局部中心性
class LocalCentrality {
 public:
  // 度中心性
  Result<DegreeCentrality> DegreeCentrality(VertexId v);
  
  // 局部聚类系数
  Result<double> ClusteringCoefficient(VertexId v);
  
  // 局部 PageRank (1-2 hop子图)
  Result<double> LocalPageRank(VertexId v, uint32_t hops);
};

// 邻居分析
class NeighborhoodAnalysis {
 public:
  // 共同邻居
  Result<std::vector<VertexId>> CommonNeighbors(
      VertexId a, VertexId b);
  
  // Jaccard 相似度
  Result<double> JaccardSimilarity(VertexId a, VertexId b);
  
  // 三角形计数 (局部)
  Result<uint64_t> LocalTriangleCount(VertexId v);
};

}  // namespace cedar::analytics::embedded
```

#### 3.2.2 分布式计算引擎 (Distributed Analytics)

```cpp
// 适用于重量级、全图分析
// 执行位置: 独立计算集群 (或预留的计算节点)

namespace cedar::analytics::distributed {

// 作业管理
class AnalyticsJobManager {
 public:
  // 提交分析作业
  Result<JobId> SubmitJob(const AnalyticsJob& job);
  
  // 查询作业状态
  Result<JobStatus> GetJobStatus(JobId id);
  
  // 获取结果
  Result<AnalyticsResult> GetResult(JobId id);
  
  // 取消作业
  Status CancelJob(JobId id);
};

// 支持的算法
enum class Algorithm {
  // 中心性
  PAGERANK,           // PageRank
  BETWEENNESS,        // 介数中心性
  CLOSENESS,          // 接近中心性
  EIGENVECTOR,        // 特征向量中心性
  
  // 社区发现
  LPA,                // 标签传播
  LOUVAIN,            // Louvain算法
  WCC,                // 弱连通分量
  SCC,                // 强连通分量
  K_CORE,             // K-Core分解
  
  // 路径与连通性
  SSSP,               // 单源最短路径
  APSP,               // 全对最短路径
  
  // 图特征
  TRIANGLE_COUNT,     // 三角形计数
  GRAPHLETS,          // Graphlet统计
  
  // 相似度
  SIMRANK,            // SimRank
  NODE2VEC,           // Node2Vec嵌入
  
  // 自定义
  CUSTOM_PREGEL,      // 自定义Pregel算法
  CUSTOM_GIRAPH,      // 自定义Giraph算法
};

// PageRank 配置示例
struct PageRankConfig {
  double damping_factor = 0.85;
  uint32_t max_iterations = 100;
  double tolerance = 1e-6;
  bool weighted = false;
  Direction direction = Direction::OUT;  // 出边/入边/双向
};

}  // namespace cedar::analytics::distributed
```

## 第四部分: 实施路线图

### 4.1 分阶段实施

```
Phase 1: 嵌入式分析 (MVP)
─────────────────────────────────────────
时间: 2-3周
目标: 支持轻量分析

功能:
- 最短路径 (BFS/Dijkstra)
- 邻居遍历 (1-3 hop)
- 简单统计 (度、三角形)

技术点:
- 复用现有存储层
- 单线程/小内存限制
- 同步API

Phase 2: 查询优化器
─────────────────────────────────────────
时间: 2周
目标: 智能路由

功能:
- 代价模型
- 轻量/重度自动判断
- 执行计划生成

Phase 3: 分布式计算框架
─────────────────────────────────────────
时间: 4-6周
目标: 全图分析能力

功能:
- PageRank
- 社区发现 (LPA/Louvain)
- 连通分量
- BSP执行引擎

技术点:
- 独立计算节点
- 消息传递
- 容错机制

Phase 4: 高级算法
─────────────────────────────────────────
时间: 4周
目标: 丰富算法库

功能:
- 图嵌入 (Node2Vec)
- 相似度计算
- GNN特征工程

Phase 5: 性能优化
─────────────────────────────────────────
时间: 持续
目标: 生产级性能

优化点:
- 缓存优化
- 并行度调优
- 内存管理
```

### 4.2 风险评估

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| 资源竞争导致TP性能下降 | 中 | 高 | 资源隔离、限流、夜间调度 |
| 大图无法内存容纳 | 高 | 中 | 磁盘迭代、分块处理 |
| 算法收敛慢 | 中 | 中 | 提前终止、近似计算 |
| 数据一致性问题 | 低 | 高 | MVCC快照、只读分析 |

## 第五部分: 需要讨论的问题

### 问题列表

1. **优先级问题**
   - 我们最迫切需要哪些图算法？
   - PageRank vs 最短路径 vs 社区发现？

2. **架构选择**
   - 是否接受独立计算节点的运维成本？
   - 还是坚持纯嵌入式 (简化运维)？

3. **资源分配**
   - 分析任务最多可占用多少CPU/内存？
   - 是否预留专用节点？

4. **接口设计**
   - 优先Cypher扩展还是新API？
   - 同步 vs 异步接口？

5. **数据新鲜度**
   - 分析结果可接受多久的延迟？
   - 需要实时更新还是离线计算？
