# CedarGraph 采用 Nebula 设计决策总结

## 决策矩阵

| 设计 | 是否采用 | 优先级 | 原因 |
|------|---------|--------|------|
| MetaD (元数据服务) | ✅ 是 | 🔴 **最高** | 分布式路由的基础 |
| Part 映射管理 | ✅ 是 | 🔴 **最高** | 与 MetaD 一起必需 |
| 负载均衡 | ✅ 是 | 🟡 中 | 运维自动化需要 |
| 存算分离 | ⚠️ 可选 | 🟢 低 | 当前非瓶颈 |
| 多 RocksDB | ❌ 否 | - | 资源开销大，不如共享 LSM |
| 弱事务 | ❌ 否 | - | Cedar 的核心优势不能丢 |

## 为什么 MetaD 优先级最高？

```
没有 MetaD 的问题:

Client 想写入 Key(entity=100, ts=now)
     │
     ├──► 问 Node0?  "我不知道 P0 在哪"
     ├──► 问 Node1?  "P0 不在我这"
     ├──► 问 Node2?  "我是 P0 Leader!"
     │
     └── 3 次网络往返才能找到正确的节点

有 MetaD:

Client ──► MetaD (缓存) ──► 直接知道 P0 在 Node2
     │
     └── 1 次查询（后续直接路由）
```

## 实施顺序

```
Phase 1 (现在): MetaD + Part 映射
   ├── 3-5 节点 Raft 元数据服务
   ├── 分区表管理
   └── 客户端路由缓存

Phase 2 (1-2月后): 负载均衡
   ├── 自动分区迁移
   └── Leader 均衡

Phase 3 (未来): 存算分离（如需要）
   └── 可选的 Query Engine 层
```

## 关键设计保留

| Cedar 优势 | 保留原因 |
|-----------|---------|
| **DTx 强事务** | 核心差异化竞争力 |
| **VSL 存储** | 时序性能优于 RocksDB |
| **共享 LSM** | 资源利用率更高 |
| **TW-CD/TW-OCC** | 创新的时序事务优化 |
